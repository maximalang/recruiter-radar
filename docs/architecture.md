# Архитектура MVP

## Цель
Сервис каждый день находит компании с сигналами найма, считает score и отправляет лиды в Telegram.

## Источники данных
1. hh.ru API — основной источник вакансий
2. ФНС / ЕГРЮЛ — позже, для обогащения компании
3. Сайт компании — позже, для контактов и контекста

## Основные части системы

### 1. Data layer
- Postgres
- Таблицы:
  - users
  - orgs
  - subscriptions
  - leads
  - signals
  - lead_status
  - deliveries

### 2. Orchestration
- n8n
- Workflow:
  1. Забрать вакансии
  2. Нормализовать данные
  3. Сохранить в БД
  4. Посчитать score
  5. Выбрать top лиды
  6. Отправить в Telegram
  7. Записать факт доставки

### 3. Backend
- Next.js / Node.js API
- Нужен для:
  - webhook’ов
  - авторизации
  - админки
  - служебных API

### 4. Telegram bot
- Получает дайджест
- Показывает карточки лидов
- Принимает действия по кнопкам:
  - contacted
  - replied
  - won
  - badfit
  - snooze

## Логика MVP

### Поток данных
1. hh API отдает вакансии
2. Сервис сохраняет вакансии и компании
3. Считаются сигналы:
   - рост вакансий
   - массовый найм
   - новые HR-роли
   - мультигород
4. Считается итоговый score
5. Лучшие лиды отправляются в Telegram

## Что делает LLM
LLM используется только для top-N лидов:
- написать 2 причины “почему сейчас”
- сгенерировать короткий opener

## Что не делаем сейчас
- сложный CRM
- много источников сразу
- сложную аналитику
- массовый скрейпинг
- генерацию текста для всех лидов

## Первый релиз
MVP должен уметь:
1. Получать вакансии из hh
2. Сохранять их в Postgres
3. Считать простой score
4. Отправлять лиды в Telegram
5. Менять статус лида по кнопке
